1 Bức tranh CI/CD tổng thể

CI/CD là "dây chuyền tự động" đưa code từ máy lập trình viên ra tới môi trường chạy thật. Chia làm hai nửa:

  • CI — Continuous Integration (Tích hợp liên tục): mỗi lần push code → tự động build, test, quét bảo mật, đóng gói. Đây là sân của Jenkins.
  • CD — Continuous Delivery/Deployment (Giao/triển khai liên tục): lấy sản phẩm đã đóng gói → đưa lên môi trường chạy (K8s, server...). Đây là sân của Argo CD.
Lập trình viên │ git push ▼ ┌──────────────────────────────┐ ┌──────────────────────────────┐ │ CI — JENKINS │ │ CD — ARGO CD │ │ │ │ │ │ 1. Lấy code (git) │ │ 5. Theo dõi Git repo cấu hình│ │ 2. Build (mvn/npm/docker) │ │ 6. So sánh mong muốn vs thực │ │ 3. Test + quét bảo mật │─────►│ 7. Đồng bộ (sync) vào K8s │ │ 4. Push image → Registry │ đẩy │ 8. Tự sửa nếu bị lệch (drift)│ │ & cập nhật manifest Git │image │ │ └──────────────────────────────┘ └──────────────────────────────┘ (làm ra "cái gì") (đưa "cái đó" đi đâu, chạy sao)
Ý chính cần nhớ Jenkins trả lời câu hỏi "code này có build/test được không, đóng gói thành gì?" — Argo CD trả lời câu hỏi "trạng thái mong muốn trong Git đã khớp với những gì đang chạy trên cluster chưa?". Hai công cụ bổ sung nhau, không thay thế nhau.

2 Jenkins là gì & kiến trúc

Jenkins là máy chủ tự động hóa (automation server) mã nguồn mở, viết bằng Java, ra đời từ 2011 (tách từ Hudson). Nó là "công cụ CI kinh điển" — cực kỳ linh hoạt nhờ hệ sinh thái 1.800+ plugin, chạy được gần như mọi việc: build, test, deploy, cron job...

Kiến trúc Controller – Agent

┌───────────────────────────┐ │ JENKINS CONTROLLER │ (trước gọi là "master") Web UI ◄────►│ - Giao diện web + API │ Git hook ────►│ - Lập lịch job │ │ - Quản lý plugin, cấu hình│ │ - Lưu kết quả build │ └───────────┬───────────────┘ │ giao việc (SSH / JNLP / inbound) ┌──────────────┼──────────────┐ ▼ ▼ ▼ ┌──────────┐ ┌──────────┐ ┌──────────┐ │ Agent 1 │ │ Agent 2 │ │ Agent 3 │ (node/worker thực thi) │ Linux │ │ Windows │ │ Docker │ └──────────┘ └──────────┘ └──────────┘
Thành phầnVai trò
Controller"Bộ não": chạy giao diện web, lập lịch, phân phối công việc, lưu cấu hình & lịch sử build. Không nên để controller tự build (rủi ro bảo mật + quá tải).
Agent (Node)"Tay chân": nơi thực sự chạy các bước build/test. Có thể là máy Linux/Windows/Mac, container Docker, hoặc pod Kubernetes sinh ra theo yêu cầu rồi tự hủy.
ExecutorMột "khe" chạy job trên agent. 1 agent có N executor = chạy song song N job.
Job / PipelineĐịnh nghĩa công việc cần làm. Hiện đại nhất là Pipeline viết bằng file Jenkinsfile.
PluginMở rộng chức năng: Git, Docker, Kubernetes, Slack, SonarQube, Trivy... Đây là sức mạnh & cũng là điểm yếu bảo mật của Jenkins.

3 Cách chạy Jenkins

Cách nhanh nhất — Docker (khuyên dùng để học)

# 1. Tạo volume để GIỮ dữ liệu (không mất khi tắt container — khác với --rm bạn từng dùng!)
docker volume create jenkins_home

# 2. Chạy Jenkins LTS
docker run -d --name jenkins \
  -p 8080:8080 \                        # cổng Web UI
  -p 50000:50000 \                      # cổng để agent kết nối vào
  -v jenkins_home:/var/jenkins_home \   # gắn volume lưu cấu hình + job
  jenkins/jenkins:lts

# 3. Lấy mật khẩu admin ban đầu
docker exec jenkins cat /var/jenkins_home/secrets/initialAdminPassword

Mở trình duyệt vào http://localhost:8080 → dán mật khẩu → cài "Suggested plugins" → tạo tài khoản admin. Xong.

⚠️ So với lệnh EJBCA trước đây của bạn Lệnh EJBCA bạn dùng có --rm nên tắt là mất sạch. Với Jenkins tuyệt đối nên dùng -v jenkins_home:... và bỏ --rm, nếu không mọi job và cấu hình sẽ bay hết mỗi lần restart.

4 Jenkins hoạt động thế nào (vòng đời một build)

  1. Kích hoạt (Trigger): có gì đó khởi động job — push code lên Git (webhook), theo lịch (cron), thủ công, hoặc job khác gọi tới.
  2. Cấp agent: controller tìm một agent rảnh (hoặc tạo pod/container mới) để chạy.
  3. Checkout: agent tải code từ Git về workspace.
  4. Chạy các stage: lần lượt Build → Test → Scan → Package theo đúng file Jenkinsfile.
  5. Artifact & báo cáo: lưu sản phẩm (image, .jar...), báo cáo test, gửi thông báo (Slack/email).
  6. Dọn dẹp: giải phóng agent; nếu là pod thì hủy luôn.
Webhook vs Polling Webhook (khuyên dùng): GitHub/GitLab "gõ cửa" báo Jenkins ngay khi có push → phản ứng tức thì. Polling: Jenkins tự hỏi Git định kỳ ("có gì mới không?") → tốn tài nguyên, chậm hơn.

5 Jenkinsfile & Pipeline

Thực hành hiện đại là "Pipeline as Code": mô tả toàn bộ quy trình trong một file Jenkinsfile đặt ngay trong repo (versioned cùng code). Có 2 cú pháp: Declarative (dễ đọc, khuyên dùng) và Scripted (linh hoạt hơn, dùng Groovy thuần).

Ví dụ Jenkinsfile Declarative — pipeline CI cho app Docker

pipeline {
  agent any                          // chạy trên agent bất kỳ

  environment {
    IMAGE = "registry.example.com/myapp"
    TAG   = "${env.BUILD_NUMBER}"   // tag theo số build
  }

  stages {
    stage('Checkout') {
      steps { checkout scm }         // lấy code
    }

    stage('Build') {
      steps { sh 'mvn -B clean package' }
    }

    stage('Unit Test') {
      steps { sh 'mvn test' }
      post { always { junit 'target/surefire-reports/*.xml' } }
    }

    stage('Security Scan (SAST)') {   // góc DevSecOps
      steps { sh 'trivy fs --exit-code 1 --severity HIGH,CRITICAL .' }
    }

    stage('Build & Push Image') {
      steps {
        sh "docker build -t ${IMAGE}:${TAG} ."
        sh "docker push ${IMAGE}:${TAG}"
      }
    }

    stage('Update GitOps repo') {   // bàn giao cho Argo CD!
      steps {
        sh """
          git clone https://git/example/k8s-manifests.git
          cd k8s-manifests
          sed -i 's|image:.*|image: ${IMAGE}:${TAG}|' deployment.yaml
          git commit -am 'Deploy ${TAG}' && git push
        """
      }
    }
  }

  post {
    failure { echo 'Build hỏng — gửi cảnh báo Slack' }
  }
}
✅ Chú ý stage cuối Jenkins không tự deploy lên K8s. Nó chỉ cập nhật số phiên bản image vào Git repo chứa manifest. Việc deploy để Argo CD lo — đây chính là điểm nối giữa hai công cụ (xem mục 9).

6 Argo CD & GitOps là gì

Argo CD là công cụ Continuous Delivery theo mô hình GitOps, chạy bên trong Kubernetes. Nó thuộc dự án Argo của CNCF.

GitOps — tư tưởng cốt lõi

GitOps quy định: Git là nguồn chân lý duy nhất (single source of truth) cho trạng thái hệ thống. Bạn không "gõ lệnh deploy" nữa — bạn sửa file trong Git, và một tác nhân (Argo CD) sẽ tự động làm cho cluster khớp với Git.

  • Declarative: mô tả "trạng thái mong muốn" (YAML), không mô tả "các bước làm".
  • Versioned trong Git: mọi thay đổi đều có lịch sử, review qua Pull Request, rollback = git revert.
  • Tự động đồng bộ (Pull-based): Argo CD nằm trong cluster, chủ động kéo cấu hình từ Git về (khác Jenkins đẩy vào — xem box dưới).
  • Tự chữa lành (Self-healing): ai đó lỡ sửa tay trên cluster → Argo CD phát hiện "lệch" (drift) và kéo về đúng như Git.
Push vs Pull — vì sao GitOps an toàn hơn Kiểu Push cổ điển (Jenkins tự kubectl apply): CI cần giữ credential quyền cao vào cluster → nếu Jenkins bị hack, cluster bị hack. Kiểu Pull của Argo CD: agent nằm sẵn trong cluster tự kéo cấu hình ra → không cần lộ credential cluster ra ngoài, an toàn hơn nhiều.

7 Cách chạy Argo CD

Argo CD cần một cluster Kubernetes. Để học, dùng minikube, kind hoặc Docker Desktop bật Kubernetes.

# 1. Tạo namespace và cài Argo CD
kubectl create namespace argocd
kubectl apply -n argocd -f \
  https://raw.githubusercontent.com/argoproj/argo-cd/stable/manifests/install.yaml

# 2. Chờ các pod chạy xong
kubectl get pods -n argocd -w

# 3. Mở giao diện web ra ngoài (port-forward tạm)
kubectl port-forward svc/argocd-server -n argocd 8080:443

# 4. Lấy mật khẩu admin ban đầu (user: admin)
kubectl -n argocd get secret argocd-initial-admin-secret \
  -o jsonpath="{.data.password}" | base64 -d

Truy cập https://localhost:8080. Có thể dùng cả Web UI lẫn CLI:

# Cài CLI rồi đăng nhập
argocd login localhost:8080
argocd app list
argocd app sync myapp

8 Argo CD hoạt động thế nào

Đối tượng trung tâm: Application

Bạn khai báo một CRD Application nói cho Argo CD biết: lấy manifest ở Git repo nào → triển khai vào cluster/namespace nào.

apiVersion: argoproj.io/v1alpha1
kind: Application
metadata:
  name: myapp
  namespace: argocd
spec:
  project: default
  source:                                # TRẠNG THÁI MONG MUỐN ở đâu
    repoURL: https://git/example/k8s-manifests.git
    targetRevision: main
    path: myapp                            # thư mục chứa YAML
  destination:                           # triển khai ĐI ĐÂU
    server: https://kubernetes.default.svc
    namespace: production
  syncPolicy:
    automated:                           # tự động đồng bộ
      prune: true                         # xóa resource không còn trong Git
      selfHeal: true                      # tự sửa khi bị chỉnh tay (drift)

Vòng lặp đồng bộ (Reconciliation loop)

┌──────────────────────────────────────────────┐ │ ARGO CD (trong cluster) │ │ │ Git ──┤ 1. Đọc "trạng thái mong muốn" từ Git repo │ repo │ 2. Đọc "trạng thái thực tế" từ K8s API │ │ 3. So sánh (diff) │ │ ├─ Synced → không làm gì │ │ └─ OutOfSync → 4. Áp dụng để khớp lại │──► K8s │ 5. Lặp lại liên tục (mặc định ~3 phút / webhook)│ cluster └──────────────────────────────────────────────┘
Trạng tháiNghĩa là
SyncedCluster khớp đúng với Git. ✅
OutOfSyncGit và cluster đang lệch nhau (có commit mới, hoặc ai đó sửa tay).
Healthy / DegradedSức khỏe thực tế của resource (pod chạy ổn hay đang lỗi).

9 Kết hợp Jenkins + Argo CD (mô hình thực tế)

Đây là kiến trúc CI/CD phổ biến nhất hiện nay, dùng 2 repo:

┌── Repo APP (code) ──┐ ┌── Repo CONFIG (manifest K8s) ──┐ │ src/, Dockerfile, │ │ deployment.yaml, service.yaml │ │ Jenkinsfile │ │ (image: myapp:123) │ └─────────┬───────────┘ └───────────┬────────────────────┘ push │ │ Argo CD theo dõi repo này ▼ ▼ ┌─────────────┐ build+test+scan ┌──────────────┐ sync ┌───────────┐ │ JENKINS │──► push image ───►│ REGISTRY │ │ K8S │ │ (CI) │ └──────────────┘ │ CLUSTER │ │ │── sửa tag image ─► Repo CONFIG ──► ARGO CD ─► deploy ─►│ └─────────────┘ (git commit) (CD, pull) └───────────┘
  1. Dev push code → Jenkins build, test, quét bảo mật, đóng gói image và đẩy lên Registry.
  2. Jenkins commit tag image mới vào repo config (manifest).
  3. Argo CD phát hiện repo config đổi → tự kéo về → deploy vào Kubernetes.
  4. Muốn rollback? Chỉ cần git revert trên repo config — Argo CD tự đưa cluster về phiên bản cũ.
✅ Phân vai rõ ràng Jenkins làm "cái gì" (artifact), Argo CD làm "trạng thái nào đang chạy". Jenkins không cần quyền vào cluster → giảm rủi ro. Git là nhật ký deploy đầy đủ, có thể audit.

10 Góc nhìn DevSecOps

🔧 Bảo mật Jenkins

  • Đừng để controller tự chạy build; tách agent riêng.
  • Quản lý bí mật bằng Credentials Plugin / HashiCorp Vault — không hardcode trong Jenkinsfile.
  • Bật RBAC (Role-based Authorization), tắt đăng nhập ẩn danh.
  • Cập nhật plugin thường xuyên — plugin là bề mặt tấn công lớn nhất của Jenkins.
  • Nhúng bước quét: SAST (SonarQube), SCA (Trivy/Snyk), secret scan (gitleaks) ngay trong pipeline.

🚀 Bảo mật Argo CD

  • Mô hình pull → không lộ credential cluster ra CI. ✔
  • Bật SSO + RBAC, đổi mật khẩu admin mặc định ngay.
  • Dùng AppProject giới hạn repo/namespace/cluster mà mỗi team được deploy.
  • selfHeal: true chống chỉnh sửa trái phép trực tiếp trên cluster.
  • Ký & xác minh commit (GPG) để chống inject manifest độc hại vào Git.
  • Mọi thay đổi phải qua Pull Request review — Git trở thành chốt kiểm soát & audit trail.
Shift-Left Triết lý DevSecOps: đẩy kiểm tra bảo mật về sớm nhất có thể. Trong mô hình này, bảo mật được "nhúng" vào pipeline Jenkins (quét sớm) và vào quy trình review Git của Argo CD (kiểm soát trước khi deploy).

Câu hỏi thường gặp khi phỏng vấn

Q: Jenkins và Argo CD khác nhau chỗ nào?

Jenkins là CI đa năng (build/test/đóng gói, mô hình push). Argo CD là CD chuyên GitOps cho K8s (mô hình pull, Git là nguồn chân lý). Thường dùng chung.

Q: GitOps là gì và lợi ích?

Dùng Git làm nguồn chân lý duy nhất cho trạng thái hệ thống; deploy = sửa Git. Lợi ích: audit đầy đủ, rollback dễ (git revert), self-heal, không lộ credential cluster ra CI.

Q: Vì sao mô hình pull (Argo CD) an toàn hơn push?

Vì agent nằm sẵn trong cluster tự kéo cấu hình, không phải đưa credential quyền cao của cluster ra hệ thống CI bên ngoài — giảm bề mặt tấn công.

Q: Controller và Agent trong Jenkins?

Controller là bộ não (UI, lập lịch, phân việc); Agent là nơi thực thi build. Tách ra để bảo mật và mở rộng.

Q: selfHeal và prune trong Argo CD?

selfHeal: tự sửa lại khi cluster bị chỉnh tay lệch Git. prune: xóa resource đã bị bỏ khỏi Git để cluster khớp đúng khai báo.
